How route-shot works intro

Three pieces, all Node. All communicate via the filesystem and a local HTTP server.

┌──────────────────────────────────────────────────────────────┐ │ menu.sh (bash launcher — install, run, open dashboard) │ └──────────┬──────────────────────────────────┬────────────────┘ │ spawns │ spawns ┌────────▼──────────┐ ┌────────▼────────────┐ │ src/route-shot.js │ │ src/server.js │ │ CLI crawler │ │ HTTP dashboard │ │ + Playwright │ │ (zero deps) │ └───────┬───────────┘ └─────┬───────────┬───┘ │ writes │ serves │ spawns ▼ ▼ ▼ ┌──────────────────────┐ ┌────────────────┐ ┌──────────┐ │ screenshots/<app>/ │◄────────►│ web/ui.html │ │ ffmpeg │ │ + index.json │ │ web/slideshow │ │ (mp4 mux)│ └──────────┬───────────┘ │ web/promo │ └──────────┘ │ read by └────────────────┘ ▼ ┌──────────────────────┐ │ videos/<run>/*.mp4 │ │ galleries/<run>/*.html │ └──────────────────────┘

Data flow

  1. Configure — apps.json (file-based) or the dashboard form (which writes to apps.json via /api/apps).
  2. Crawl — route-shot.js launches Playwright Chromium, BFS-walks same-origin links, runs preSteps + dismiss + clicks + autoButtons per page, saves .png + index.json.
  3. Render — server.js opens slideshow.html / promo.html in Playwright's recordVideo context (webm), then hands off to ffmpeg for mp4 muxing + audio overlay.

Which click mode should I use? decision tree

route-shot has three overlapping ways to interact with a page: preSteps, clicks, autoButtons. Pick by intent:

What do you want to capture? │ ├─ Just the landing page, no interaction ─────────────► maxPages:1, nothing else │ ├─ Same URL, several UI states │ │ │ ├─ I know the exact buttons │ │ │ │ │ ├─ Tab-like (state should persist) ───────────► clicks + clickMode:sequential │ │ └─ Each click independent of the others ───────► clicks + clickMode:independent │ │ │ └─ I don't — just click everything reasonable ───► autoButtons:true + blocklist │ ├─ A whole site (many URLs) ──────────────────────────► default BFS (no clicks needed) │ └─ Setup first, then capture │ ├─ Setup is "one-shot" (login, toggle) ──────────► preSteps[] └─ Setup + then click each tab ──────────────────► preSteps[] + clicks[]

clicks[]

  • Explicit, deterministic list
  • Survives across runs
  • You own the selectors
  • Best for tab-navigation snapshots

autoButtons:true

  • Zero-config discovery
  • Blocklist keeps destructive actions safe
  • Can produce near-duplicates
  • Best for initial surveys

preSteps[]

  • Ran before screenshots
  • Runs once per independent reload
  • Chains fill → click → wait
  • Best for login / onboarding gates

🚀 Start here

New to route-shot? Follow these five steps — each one is what you'd do right after the previous.

1

Install once

Clone the repo and run the installer. Requires Node 18+.

git clone https://github.com/abourdim/route-shot.git
cd route-shot
./menu.sh          # pick option 2

Installs Playwright + Chromium. Takes ~60 s.

2

Launch the dashboard

The dashboard is the friendliest way to use route-shot — no JSON, no terminal after this.

./menu.sh          # pick option 6
# or:
node src/server.js

Opens http://localhost:8080. If taken, auto-bumps to 8081, 8082…

3

Pick or edit an app

Use the App editor panel (top-right). Choose one of the presets, or click + New and fill in name + url.

Use the Quick recipes chips to apply a proven config in one click: Only main page, Dismiss common modals, Auto-snap every button, …

The sticky header shows which app you're editing so you never lose track.

4

Capture

Hit Run batch (apps.json) — the status pill turns amber and logs stream.

Or try Manual Snapshot Session: opens a headful Chromium; a 📸 widget injects into the page. Drive by hand, snap on demand. Hotkey: Ctrl+Shift+S.

Not sure which buttons to click? Scan URL lists every clickable element on the page so you can pick the ones you want.

5

Review and export

Screenshots appear right below Run. Click any thumbnail for a lightbox.

Then in the Export section, pick a tab:

  • 🖼️ Promo PNG — social-media card (Twitter, IG, YouTube, …)
  • 📄 Gallery + PDF — self-contained HTML handout
  • 🎬 Slideshow video — MP4 with optional music + fades
That's it. Come back to this guide for the dashboard tour, field reference, or when something surprises you.

Dashboard tour

The dashboard is a single page served by node src/server.js. Six panels, top-down:

Sticky header
Stays pinned while you scroll. Shows editing: <app-name> → <url> so you always know which apps.json entry is active. A pill on the right reports status: gray "idle" · amber "running — batch" · green post-run with last exit code.
RUN panel
Single-URL crawler, plus Run batch (apps.json) / Dry run / Stop / Clean screenshots (destructive, pushed to the right) / Open folder. Live logs stream into the console area.
APP EDITOR
Form-based apps.json editor. Accordions for Essentials, Page & navigation, Auto-discover buttons, Setup steps (preSteps), Explicit clicks, Timing & debug, plus a Raw JSON & presets section for power users. Each accordion has a badge showing current state.
Save to apps.json · Save as preset · Dry run this app. Uploads and imports from DevTools Recorder live here too.
Screenshots
Grid of every file under screenshots/. Click any thumbnail for the lightbox (scrollable full-res, open-in-tab, delete). Top strip summarizes each index.json (page count · ok · skipped · errors), with View JSON for the raw schema.
MANUAL SNAPSHOT SESSION
Opens a headful Chromium you drive by hand. A draggable 📸 widget injects into the page; hit Ctrl+Shift+S or the widget to snap. Great for hardware pairing, login flows, anything too tricky to automate.

Takes a URL, opens it in a background Chromium, returns every button / [role=button] / input[type=button] / input[type=submit]. Click-to-copy cells for label, id, and selector. Filter box + Add to clicks / Add to blocklist buttons pipe straight into the app editor.

Three tabs: Promo PNG (a framed social card), Gallery + PDF (self-contained HTML handout + optional printed PDF), and Slideshow video (reels/feeds with optional audio). See the export section for details.

Task recipes

Pick the card that matches what you want. Each has the exact steps.

📸 One screenshot of a page

  1. Open the dashboard.
  2. In App editor, click + New, fill name + url.
  3. Click Only main page recipe (sets maxPages: 1).
  4. Save to apps.json → Run batch.

🔘 Every button auto-snapped

  1. Only main page, then Auto-snap every button.
  2. If the page opens with a modal, set Dismiss selector (e.g. button:has-text("Let's Go")).
  3. Tune autoButtonsBlocklist if some buttons shouldn't be clicked.
  4. Run batch. Variants appear as ..._v02_<label>.png.

🔐 App behind login

  1. Use the Setup steps (preSteps) accordion.
  2. Add fill for email + password, click the sign-in button, then waitForURL the post-login route.
  3. Session cookies persist for the whole crawl.

Alternatively record the flow in Chrome DevTools Recorder and import the JSON (§ Recorder import).

📑 SPA tabs (one screenshot per tab)

  1. Run Scan URL on the page.
  2. In the results, click the ID cells for each tab to copy. Paste into Explicit clicks.
  3. Set clickMode: independent (reloads between each) — so every screenshot is a fresh tab.

🎬 Social media video

  1. Scroll to Export → 🎬 Slideshow video.
  2. Pick format (Reel / Feed / YouTube / Square) and a style (32 effects across Clean, Tech, Space, Retro).
  3. Click shot thumbnails to include, or leave empty for all.
  4. Optional: pick an audio track, set volume + fade. Watch the live pre-flight read back what's about to render.
  5. Render video. ffmpeg muxes audio into mp4.

🆚 Compare 3–5 video styles

  1. Same shots — but pick multiple cards in the Test modes grid.
  2. Render batch. Writes one mp4 per style into a timestamped run folder.
  3. A compare.html is generated automatically so you can preview them side-by-side.

🖐️ Drive by hand (hardware pairing, etc.)

  1. In Manual Snapshot Session, set URL + app name.
  2. Click Start session — a headful Chromium opens with a 📸 widget.
  3. Drive the page manually. Snap with Ctrl+Shift+S or the widget.
  4. Filenames include your label. Session state (label) persists across reloads.

📤 Share a single promo

  1. Pick a screenshot in Export → 🖼️ Promo PNG.
  2. Choose size (Twitter, IG, YouTube, Product Hunt, …), frame (browser / laptop / phone / none).
  3. Title + tagline + CTA + accent color → Render PNG.

Quick recipes (dashboard buttons)

Five chips under the app editor apply proven configs in one click. They only mutate the form — press Save to apps.json to persist.

ChipWhat it changes
Only main pageSets maxPages: 1. No BFS — snapshot the landing URL (plus any variants/clicks) only.
Dismiss common modalsIf dismiss is empty, sets it to button:has-text("Let's Go"), button:has-text("Got it"), button:has-text("Accept").
Auto-snap every buttonEnables autoButtons: true. If blocklist is empty, seeds the standard safety list (delete|remove|clear|reset|logout|pay|confirm|send).
Isolated shots (independent)Enables autoButtons + sets clickMode: independent — reloads between every click, so every variant is from a fresh state.
Headful debugSets headful: true. Chromium window opens so you can see what's happening.

Presets subsystem

A preset is a saved apps.json under presets/. The dashboard has a picker that lists every file there.

How to use

  • Save — click Save as… in the app editor (or in the Raw JSON & presets accordion), give a name. Writes presets/<name>.json.
  • Load — pick from the dropdown, click Load. Replaces the current apps.json in the form (not saved yet until you click Save).
  • Auto-load — check the auto-load box to apply the preset every time you change the dropdown. Great for switching configs fast.
  • CLI — node route-shot.js --preset <name> runs any preset directly.
  • Delete — the Delete button removes presets/<name>.json from disk.

Bundled presets

Fourteen presets ship with the repo — bit-playground, debug, docs-site, github-pages-portfolio, hardware-pairing, landing-page, login-flow, minimal, modal-stress, spa-tabs, etc. Browse presets/ in the repo for concrete real-world configs.

Manual Snapshot Session

The dashboard's third panel opens a headful Chromium you drive by hand, with a floating 📸 snap widget injected into every page.

Purpose

Some flows can't be automated: hardware pairing dialogs, payment forms, OTP entry, user testing. Manual session lets you drive the page yourself while still producing deterministic, labelled screenshots.

How it works

  1. Fill URL + app name (becomes the output subfolder).
  2. Click Start session. Chromium opens headful.
  3. A draggable 📸 widget appears bottom-right of every page the browser visits.
  4. Type a label in the optional snapshot label field (persists across reloads via localStorage).
  5. Hit Ctrl+Shift+S, or the widget, or the dashboard's Snapshot button — all three paths queue a snap. The widget is hidden for the frame so it doesn't appear in the screenshot.
  6. Click Stop to close the browser.

Behind the scenes

  • Page-level queue — the widget posts to a queue; Playwright drains it and calls page.screenshot. Avoids same-origin fetch issues (which HTTPS contexts block when calling localhost).
  • CORS is wide-open on /api/manual/snapshot so the widget works on any domain.
  • File naming: manual/<app>_<n>_<label>.png.

Scan URL → Pick buttons

Given a URL, Playwright opens it, runs dismiss if set, and enumerates every clickable-looking element.

Columns

ColumnMeaning
Tagbutton, a, input[type=button], [role=button]
LabelVisible text. Clickable — copies to clipboard.
ID#element-id if set. Clickable — copies.
SelectorBest stable CSS selector. Clickable — copies.
VisibleHidden elements are dimmed.

Actions

  • Filter box — incremental match against label / id / selector.
  • Add to clicks — appends the selected row's selector to the current app's clicks[].
  • Add to blocklist — takes the label text and appends to autoButtonsBlocklist.
Use scan to discover selectors, then use Isolated shots recipe for clean variant capture.

Variant filenames — full grammar

<pageIdx>_<urlSlug>[__v<variantIdx>[_d<depth>[_<parent>__<child>]]_<label>].png

001_abourdim_github_io_bit_playground.png
 │   └─ URL with non-word chars → underscores
 └─ page index in BFS order (zero-padded)

001_abourdim_github_io_bit_playground__v02_Controls.png
 │                                       │  └─ click label
 │                                       └─ variant number (01 = main, 02+ = after click)

001_abourdim_github_io_bit_playground__v05_d1_Controls__Sensors.png
 │                                       │  │  │         └─ leaf click
 │                                       │  │  └─ ancestor click (depth-0 click that got us here)
 │                                       │  └─ depth indicator (appears when maxDepth > 0)

001_abourdim_github_io_bit_playground__v12_emoji_a3f9k2.png
                                              │     └─ 6-char hash of the emoji (for unique filename)
                                              └─ "emoji_" prefix for emoji-only labels
Whitespace, slashes, and funky unicode get squashed to underscores. For pure-emoji buttons route-shot can't safely include in a filename, it hashes the label so two emoji buttons never collide.

Audio (video export)

Videos can be rendered silent (default), or muxed with a music bed via ffmpeg.

Three ways to provide audio

  1. Library — drop .mp3 / .wav / .m4a / .ogg files into audio/library/. They appear in the dropdown.
  2. Uploads — files placed under audio/uploads/ also appear in the dropdown (useful for per-session stuff you don't want to commit).
  3. URL — paste an MP3/WAV URL. route-shot downloads it once into audio/cache/<sha1>.mp3 and reuses the cached copy on subsequent renders.

Controls

FieldRangePurpose
Volume0–100%Linear gain (scaled to 0.0–1.0 before passing to ffmpeg)
Fade-in0–∞ msSilence → full volume ramp at video start
Fade-out0–∞ msFull volume → silence ramp before video end
▶ Preview—Play the selected file / URL in a floating <audio> element right in the dashboard
🔇 Silent (batch)checkboxForce every batch render to be silent (useful for comparing visual styles without audio dominating)
The audio is looped via -stream_loop -1, so a 10-second track covers a 30-second video. Fade-out starts at videoSeconds − fadeOutMs, not at track-end.

apps.json structure

{
  "apps": [
    { /* app 1 */ },
    { /* app 2 */ }
  ]
}

Each app object must have name and url. Everything else is optional.

Strict JSON — no trailing commas, no comments. The last key in an object must not have a , after it.

Two complementary click modes

ModeUse when
clicks: [...] explicitYou know the buttons you want. Full control.
autoButtons: trueYou want every discoverable button. Zero config beyond a blocklist.

You can combine both — explicit clicks run first, then auto-discovery fills in the rest (deduped).

Chrome DevTools Recorder import

  1. DevTools → Recorder panel → Start new recording
  2. Click through the flow you want captured
  3. End recording → Export → "Export as a JSON file" (NOT Puppeteer or Playwright — those are JS scripts route-shot can't parse)
  4. Save as flow.json

Then convert:

# print the translated entry to stdout
node route-shot.js --import-recording flow.json

# or merge directly into apps.json (upsert by name)
node route-shot.js --import-recording flow.json --name my-app --merge apps.json

# treat the recorded clicks as variants (one screenshot per click) instead of one-shot preSteps
node route-shot.js --import-recording flow.json --as-clicks --merge apps.json
FlagMeaning
--import-recording <path>Source Recorder JSON
--name <name>Override app name (default: recording title)
--url <url>Override start URL
--as-clicksMap clicks to clicks[] instead of preSteps[]
--merge <apps.json>Upsert into the config file
The importer picks the most stable selector candidate from Recorder's list — prefers #id over aria/…, xpath/…, pierce/…, text/… alternatives.

autoButtonsBlocklist is a regex tested against each button's visible text. Case-insensitive. | means OR.

Performance — what's fast, what isn't

Rough timings on a 2024 desktop with Chromium + local fibre (your mileage varies).

OperationTimeNotes
One page, one screenshot~2–4 sPlaywright cold-start + goto + screenshot
BFS of a small site (10 pages)~20–40 sCold-start paid once
autoButtons, sequential, 20 buttons+10–15 sNo reload between clicks
autoButtons, independent, 20 buttons+40–80 sFull reload per click
Dry-run (plan only)< 1 s per appNo browser, just config validation
Slideshow video, 10 shots @ 2.5 s/frame~30–60 sPlaywright records 25 s webm, ffmpeg re-encodes to mp4 in <10 s
Batch render, 5 styles, 10 shots~3–5 minSequentially renders each style
Gallery HTML~1 sInlines images as data URIs — output size ≈ sum of pngs
Gallery → PDF+~5 sRouted through Playwright print-to-PDF

Ways to speed things up

  • Prefer clickMode: sequential — 5–10× faster than independent. Use independent only when state truly leaks.
  • Lower autoButtonsMax — default is 40 per page. Cap at 10 for a quick sweep.
  • Tighten maxPages — don't crawl a whole blog if you only want the product pages.
  • Batch your video styles offline — queue with Render batch, walk away.
  • Reuse the dashboard — Chromium launch is the expensive step; the 2nd run is noticeably faster.

Security & privacy

What leaves your machine

Essentially nothing. route-shot is local-first:

  • Screenshots stay on disk under screenshots/.
  • The dashboard binds to localhost by default.
  • No telemetry, no update checks, no analytics.
  • Audio URLs pasted into the video export are fetched (once, cached into audio/cache/).
  • When crawling, Playwright fetches the target site normally — same data flow as opening it in Chrome.
Don't point autoButtons at production with real data. Even with a blocklist, a mis-labelled button could trigger a "Delete account" action. Use staging. Seriously.

The dashboard has no auth

Any process on localhost can POST to /api/run/batch, /api/export/video, or /api/apps. Fine for a single-user dev machine. If you expose the port (port-forward, tunnelling tool), put a reverse-proxy with basic auth in front — don't bind directly.

ROUTE_SHOT_ALLOW_EVAL

The evaluate preStep runs arbitrary JS in the target page. Only honored when the env var is set to 1. Leave it off unless you control every apps.json entry.

FAQ

Do I need to know JavaScript?

No — the dashboard is form-driven. You'll touch JSON if you import a DevTools Recorder file or power-edit in the Raw JSON section, but that's it.

How is this different from just screenshotting the site?

A single screenshot captures one state of one page. route-shot BFS-crawls the whole site, captures every page, and optionally one screenshot per clickable button. That scales to "show me every state of every page" without writing a test.

Why Chromium and not Firefox/WebKit?

Playwright supports all three. Chromium is hard-coded here because the recordVideo codec + ffmpeg pipeline is simplest in Chromium. The crawler itself would work on any browser.

Can I screenshot an Electron / Tauri app?

Not directly — route-shot drives Chromium against a URL. If the Electron app exposes a devtools port, you can crawl that. For desktop apps, the Manual Snapshot Session + OS screenshot tools are closer.

What's the difference between dismiss and preSteps?

dismiss runs every page load (including after independent reloads) — designed for modals that pop up on arrival. preSteps runs only at the start of the app + on independent reloads that need the setup state restored. Use dismiss for "always close this modal" and preSteps for "log in and click expert mode before starting".

Do I need ffmpeg?

For video export only. If it's on PATH, you get an mp4 (with audio if supplied). Without it you get a .webm from Playwright's recordVideo. Promo PNG + Gallery HTML + PDF don't need it.

Can I crawl multi-origin apps?

BFS is restricted to the same origin as the start URL. For multi-origin flows, list them as separate apps in apps.json.

What happens if a page 404s?

Still screenshot-captured, but status: 404 is recorded in index.json. status: 0 means network failure (DNS, connection reset, timeout).

How do I share a gallery with someone?

Export → Gallery + PDF. The resulting gallery.html is fully self-contained (images are data-URI inlined). Email the file, attach to Slack, host anywhere — no dependencies.

Where do the video styles come from?

They're CSS + JS animations defined in web/slideshow.html. Playwright opens that page with your shots as config, records it playing, ffmpeg transcodes. To add a style, drop CSS into the slideshow and register it in the styles list.

Tips & gotchas

Don't crawl, just snapshot — if you only want one screenshot of the landing URL, set "maxPages": 1. Otherwise BFS follows every <a href>.
Modal blocks your preSteps? route-shot now dismisses before preSteps, but if your modal uses a different selector than the one you think, the click will silently fail. Enable "headful": true and watch it happen live.
autoButtons captures way too many near-duplicates? Many buttons on the page may only toggle a tiny visual detail. Use clicks[] with an explicit list instead.
Independent mode is expensive — reloads the page for every single click. With 40 buttons that's 40 full page loads. Budget accordingly or drop autoButtonsMax.
Playwright has-text is substring — has-text("Clear") matches both "Clear" and "Clear All". If you need exact match, use text-is("Clear") or role-based getByRole('button', { name: 'Clear', exact: true }) (autoButtons already does this).
Never run autoButtons on a production app with real data. Even with a blocklist, you can't be sure "Save" won't trigger something you don't want. Use a staging env or the default blocklist as a minimum.
preSteps don't replay after goto in BFS — they replay after every independent-mode reload (auto-button variants), but not when the BFS crawls to a new URL. Same-app multi-page flows may lose preSteps state.
Session persists — a single browser context is used for the whole app, so cookies and localStorage survive page navigations. Log in once in preSteps, crawl all pages authenticated.

Troubleshooting

"Element is not visible, enabled and stable"

A modal or overlay is covering the target. Check dismiss selector. Bump dismissWait. Try "headful": true to see it.

"Not found" / "elementHandle.click: Timeout"

Selector doesn't match anything. Two reasons:

  • CSS is case-sensitive for IDs — #ExpertBtn ≠ #expertBtn
  • Auto-discovery uses exact role-name match — if the label has weird whitespace or multiple elements share it, disambiguate with an explicit selector in clicks[]

"Unexpected token … is not valid JSON"

Usually a trailing comma. Last key in an object, last item in an array — no comma after it.

{
  "foo": 1,
  "bar": 2,   ← OK (not last)
  "baz": 3    ← no comma!
}

Importer says "looks like a Puppeteer/Playwright script export"

You exported as Puppeteer/Playwright JS instead of JSON. In the Recorder panel, pick "Export as a JSON file" (top of the list).

Top-nav tabs aren't clicked

Happens when multiple elements share the tab text (e.g. a heading plus the button). Fix: add the tab buttons to clicks[] using ID selectors like #controlsTabBtn. Auto-discovery will skip them thanks to dedup.

Welcome modal keeps coming back every screenshot

Independent mode reloads for every click, so the modal re-renders each time. Dismiss handles this. If it's still visible in screenshots, either dismissWait is too short or the selector is wrong.

Glossary

BFSBreadth-first search. route-shot's link crawling: visits the start URL, then every <a href> it finds, then theirs, and so on — level by level.
VariantA screenshot of the same URL after a click. Captured as ..._v01_<label>.png.
Independent modeEach click starts from a fresh reload. Prevents state leaking between clicks. Slower but cleaner.
Sequential modeClicks accumulate. Matches tab-like navigation where state should persist.
preStepsSetup actions (login, toggles) run once at start and replayed on every independent reload.
dismissSelector(s) clicked immediately after every page load — typically onboarding modals, cookie banners.
RecorderChrome DevTools' Recorder panel. Records user interactions, exports to JSON / Puppeteer / Playwright. route-shot only consumes the JSON.
BlocklistRegex of button texts to skip during autoButtons discovery. Prevents clicking destructive actions.
